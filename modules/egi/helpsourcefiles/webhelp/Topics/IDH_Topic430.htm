Radio waves

 

# 3 - 30 MHz. High Frequency (HF). This is also known as the Short Wave band. It includes the CB band (see below) and the channels used for radio control. As the frequency of the carrier wave increases, it becomes possible to encode more information and to crowd channels (proportionately) closer together.

# 30 - 300 MHz. Very High Frequency (VHF). (includes FM radio and television). Antennae are often made to be about one quarter or one half wavelength long.

# 300 MHz - 3 GHz. Ultra High Frequency (VHF). (GHz = 109 Hz). This includes some television and mobile phones: see below. Many channels are available.

# 3 - 30 GHz. Super High Frequency (SHF). (roughly corresponds to microwave band) Used for communication with satellites.

# 30 - 300 GHz. Extra High Frequency (EHF). Not much used for radio communication (yet), because of the technological difficulty of encoding and decoding amplitude and frequency modulation at such high frequencies.

Each Application in the eGov Suite is called a Module. Each of these modules can be subdivided into sub-modules and the Sub-modules will have a number of features.

Module/Sub module will be enabled for a ULB from this screen.

 

There can be any level of Hierarchy with in the Modules. 

 

Action Set up

A module/sub module can have multiple actions. They are the leaf level nodes in the menu tree or individual Action Classes and JSP’s. There should be an entry in the Action table for each and every JSP/HTML/Action class in the System. Each Action is Uniquely identified by it’s name. Action will always have a URL by which we can access it which can be broken into 3 components

1.                    Domain name

2.                    Base URL

3.                    Query Params

 

Eg: http://www.egovernments.org/payment/Supplier.jsp?mode=Cheque&id=238&amount=3000&suppid=430

has

Domain Name: http://www.egovernments.org

Base URL: /payment/Supplier.jsp

and the query Params: mode=Cheque&id=238&amount=3000&suppid=430

Access to the action will be controlled on Base URL and the Query Params. Multiple Actions can have the Same Base URL but the Base URL and Query Params combination must be unique.

Base URL and Query Params will be used along with Role Action mapping to allow/disallow access to a given action for a given user.

The Query Params entered in the action can be a subset of the params in the complete URL as only params required for access control are required in the Action Set up.

Role Module and Action Mapping

A Role can be given access to certain modules/actions in the system. The creation of modules/actions has been explained in the previous sections. The mapping will be used to allow access to features in the system based on the role of the logged in user.

The role can be allowed an access to an entire module in which case he will have access to all the actions under that role. Alternatively he can be allowed access to individual actions with in a given module.

 

The user can have multiple roles. In this case, if any role allows access to the action, the RBAC shall allow the access to the user.

 

The screen will be used to both view as well as editing the role to action mapping.

 

Set up Department

Each ULB has a set of Departments. The following screen models the Department in the ULB. This screen can be used to create/modify/delate the Departments. Department once created cannot be deleted if there are any users created under the

 

Create Designation and Positions

A department will have a number of designations. For each designation there will be a number of positions. Each position will have a name and a number of positions.

 

During the employee creation phase, the employee would be assigned a designation.

 

Create/Edit Users and Assign Role to Users

The user in the application can be created using this application (which will capture the bare minimum details) or the users can be created by individual applications by using the API’s provided by the User Module. Eg: PIMS module can create Employee as a user, Property Tax application can create citizen as a user and EGF can create Supplier as a user in the system. These modules will normally add more details as required by their respective modules and would be stored in the application scope.

 

The core User feature focused on the following:

a.                    Create a user with User name, First/Middle and Last names and the password.

The role of the user

The field/Boundary of the user

a.                    Making the user active/inactive and tracking the dates of active/inactive

 

Employee belongs to the department. User can belong to a department by the virtue of him being an employee.

 

User Favorites

Based on the user’s primary work, s/he normally goes through the same set of links every time and it is troublesome for him/her to go through the complete hierarchy of the menu tree. There should be a facility for him to save the list of repetitive actions as favorites and these should appear at the top just below the root menu.

 

These actins can appear again in the normal hierarchy as well.
